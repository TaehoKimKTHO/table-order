## 2. 보안 요구사항

### 2.1 인증 방식

| 대상 | 방식 | 구현 | 비고 |
|---|---|---|---|
| 고객 | 세션 토큰 | UUID v4, Authorization 헤더 또는 Cookie | Unit 1 Auth Module 활용 |
| 관리자 | 없음 (MVP) | /admin 경로 직접 접근 | 보안 확장 미적용 |

**MVP 범위 결정**: 요구사항 constraints.md에 따라 복잡한 인증(OAuth, 2FA)은 제외. 고객은 세션 토큰 기반, 관리자는 경로 기반 접근으로 단순화.

### 2.2 권한 관리

| 역할 | 주문 생성 | 주문 조회 | 상태 변경 | 주문 삭제 |
|---|---|---|---|---|
| 고객 | ✅ (자기 세션) | ✅ (자기 세션) | ❌ | ❌ |
| 관리자 | ❌ | ✅ (전체) | ✅ | ✅ |

**구현 방식**: API Route 경로 기반 분리 (/api/customer/* vs /api/admin/*)
- 고객 API: 세션 토큰 검증 미들웨어 적용
- 관리자 API: MVP에서는 별도 인증 없이 접근 허용

### 2.3 데이터 보안

| 항목 | 적용 여부 | 설명 |
|---|---|---|
| 전송 암호화 (HTTPS) | 배포 시 적용 | 로컬 개발은 HTTP |
| 저장 데이터 암호화 | 미적용 | SQLite 파일 기반, MVP 범위 외 |
| SQL Injection 방지 | ✅ 적용 | Prepared Statement 사용 (sql.js bind) |
| 입력값 검증 | ✅ 적용 | 비즈니스 규칙 VR-ORD-01 적용 |
| XSS 방지 | ✅ 적용 | React 기본 이스케이핑 + JSON 응답 |

## 3. 기술 스택

### 3.1 프로그래밍 언어 및 프레임워크

| 구분 | 선택 | 버전 | 선택 근거 |
|---|---|---|---|
| 언어 | TypeScript | ^5.7.0 | 타입 안전성, 프론트/백엔드 통합 |
| 프레임워크 | Next.js (App Router) | ^15.1.0 | 풀스택, SSR/API Routes 통합 |
| 런타임 | Node.js | 18+ | Next.js 요구사항 |
| UI 라이브러리 | React | ^19.0.0 | Next.js 기본 |

### 3.2 데이터베이스

| 구분 | 선택 | 선택 근거 |
|---|---|---|
| DBMS | SQLite (sql.js) | 단일 매장 MVP, 서버리스 배포 용이, 별도 DB 서버 불필요 |
| ORM | 없음 (Raw SQL) | 경량 MVP, sql.js 직접 사용 |
| 파일 저장 | 로컬 파일 시스템 | data/table-order.db |

**SQLite 선택 근거**:
- 단일 매장 운영 (동시 사용자 ~100명)
- 읽기 위주 워크로드 (메뉴 조회 > 주문 생성)
- 별도 DB 서버 운영 비용 없음
- WAL 모드로 읽기/쓰기 동시성 지원

### 3.3 캐싱 전략

| 항목 | 캐싱 여부 | 설명 |
|---|---|---|
| 메뉴 데이터 | 미적용 (MVP) | 변경 빈도 낮으나 SQLite 자체 성능 충분 |
| 주문 데이터 | 미적용 | 실시간성 요구, 캐시 무효화 복잡 |
| 세션 데이터 | 미적용 | DB 조회 성능 충분 |
| SSE 연결 | 인메모리 Map | 이미 구현됨 (Unit 1) |

**캐싱 미적용 근거**: 단일 매장 MVP에서 SQLite + 인덱스 조합으로 목표 응답 시간 달성 가능. 캐시 레이어 추가 시 복잡도 대비 이점 미미.
