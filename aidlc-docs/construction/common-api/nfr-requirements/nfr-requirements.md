# 테이블오더 서비스 - 비기능 요구사항

## 1. 성능 요구사항

### 1.1 예상 동시 사용자 수

| 구분 | 수치 | 근거 |
|---|---|---|
| 최대 동시 고객 수 | 10명 | 테이블 10개 이하, 테이블당 1 태블릿 |
| 최대 동시 관리자 수 | 2명 | 카운터 PC/태블릿 1~2대 |
| 총 동시 접속 | 12명 | 고객 10 + 관리자 2 |
| SSE 동시 연결 | 12개 | 접속자 수와 동일 (1인 1연결) |

### 1.2 응답 시간 목표

| API 유형 | 목표 응답 시간 | 비고 |
|---|---|---|
| 메뉴 조회 (GET) | < 200ms | 읽기 전용, 캐싱 불필요 (소규모) |
| 주문 생성 (POST) | < 500ms | DB 쓰기 + SSE 브로드캐스트 포함 |
| 주문 상태 변경 (PATCH) | < 300ms | DB 업데이트 + SSE 알림 |
| 로그인/세션 검증 | < 200ms | DB 조회 1~2회 |
| 이미지 업로드 | < 2000ms | 파일 크기 최대 5MB 기준 |
| SSE 이벤트 전달 | < 2000ms | 주문 생성 시점부터 관리자 수신까지 |

### 1.3 처리량 목표

| 지표 | 목표 | 근거 |
|---|---|---|
| 초당 주문 처리 | 5건/초 | 피크 시간 기준 (10테이블 동시 주문 가정) |
| 초당 API 요청 | 50건/초 | 메뉴 조회 + 주문 + 상태 변경 합산 |
| 일일 총 주문 수 | 500건 | 영업 10시간, 시간당 50건 |

## 2. 보안 요구사항

### 2.1 인증 방식

| 대상 | 방식 | 상세 |
|---|---|---|
| 고객 (태블릿) | Session Token | UUID v4 기반 세션 토큰, Authorization 헤더 전달 |
| 관리자 | 미적용 (MVP) | /admin 경로 직접 접근, 향후 JWT 확장 예정 |

**MVP 인증 설계:**
- 고객: `POST /api/customer/auth/login` → 세션 토큰 발급 → 이후 요청에 `Bearer {token}` 헤더 포함
- 관리자: 인증 없이 /api/admin/* 직접 접근 가능
- 세션 만료: 없음 (관리자 이용 완료 처리 시에만 세션 종료)

### 2.2 권한 관리

| 대상 | 접근 범위 | 제어 방식 |
|---|---|---|
| 고객 | /api/customer/* | 세션 토큰 검증 (Authorization 헤더) |
| 관리자 | /api/admin/* | 경로 기반 접근 (MVP — 인증 미적용) |

**고객 권한 제어:**
- 모든 고객 API 요청에 `Authorization: Bearer {sessionToken}` 필수
- 세션 토큰이 없거나 유효하지 않으면 401 Unauthorized 반환
- 고객은 자신의 세션에 속한 주문만 조회 가능 (세션 격리)
- 주문 상태 변경, 삭제 등 관리자 전용 기능 접근 불가

**관리자 권한 제어 (MVP):**
- /api/admin/* 경로에 대한 별도 인증 없음
- 향후 JWT 기반 관리자 인증 확장 예정
- 네트워크 수준 접근 제어로 보완 (매장 내부 네트워크만 허용)

### 2.3 데이터 보호

| 항목 | 적용 방식 | 비고 |
|---|---|---|
| 전송 구간 암호화 | HTTPS (TLS 1.2+) | 프로덕션 환경 필수 |
| 저장 데이터 암호화 | 미적용 (MVP) | SQLite 파일 기반, 서버 내 저장 |
| 세션 토큰 | UUID v4 (128비트 랜덤) | 예측 불가능한 토큰 생성 |
| 민감 정보 | 없음 | 개인정보 수집하지 않음 (테이블 번호만 사용) |
| SQL Injection | Prepared Statement 사용 | 모든 DB 쿼리에 파라미터 바인딩 적용 |
| XSS | React 기본 이스케이핑 | JSX 자동 이스케이핑 + CSP 헤더 |

## 3. 기술 스택

### 3.1 프레임워크 및 런타임

| 구분 | 기술 | 버전 | 선정 근거 |
|---|---|---|---|
| 런타임 | Node.js | 20 LTS | 안정성, 장기 지원 |
| 프레임워크 | Next.js (App Router) | 15.x | 풀스택 SSR/API Routes, React 19 지원 |
| UI 라이브러리 | React | 19.x | 최신 서버 컴포넌트 지원 |
| 언어 | TypeScript | 5.7+ | 타입 안전성, 개발 생산성 |

### 3.2 데이터베이스

| 구분 | 기술 | 선정 근거 |
|---|---|---|
| DBMS | SQLite (sql.js) | 서버리스, 설치 불필요, 단일 매장 규모에 적합 |
| 접근 방식 | sql.js (순수 JS) | 네이티브 빌드 도구 불필요, 크로스 플랫폼 |
| 동시성 | WAL 모드 | 읽기/쓰기 동시 처리 (최대 12명 동시 접속) |

### 3.3 실시간 통신

| 구분 | 기술 | 선정 근거 |
|---|---|---|
| 프로토콜 | Server-Sent Events (SSE) | 단방향 실시간 전송, HTTP 기반, 구현 단순 |
| 연결 관리 | 인메모리 Map | 소규모 (최대 12연결), 별도 메시지 브로커 불필요 |

### 3.4 파일 관리

| 구분 | 기술 | 선정 근거 |
|---|---|---|
| 이미지 저장 | 로컬 파일 시스템 | 단일 서버 운영, S3 등 외부 스토리지 불필요 |
| 이미지 서빙 | Next.js Static Files | public/ 디렉토리 기반 정적 파일 서빙 |
| 최대 크기 | 5MB | 메뉴 이미지 용도에 충분 |

### 3.5 컨테이너화

| 구분 | 기술 | 선정 근거 |
|---|---|---|
| 컨테이너 | Docker | 일관된 배포 환경, 의존성 격리 |
| 오케스트레이션 | Docker Compose | 단일 서비스 구성, 로컬 개발 환경 |

### 3.6 개발 도구

| 구분 | 기술 | 용도 |
|---|---|---|
| 패키지 관리 | npm | Node.js 기본 패키지 매니저 |
| 린팅 | ESLint (Next.js 기본) | 코드 품질 관리 |
| 타입 체크 | TypeScript Compiler | 컴파일 타임 타입 검증 |

## 4. 가용성 및 안정성

### 4.1 가용성 목표

| 지표 | 목표 | 비고 |
|---|---|---|
| 서비스 가용성 | 99% (영업 시간 기준) | 단일 매장, 영업 시간 내 |
| 계획 다운타임 | 영업 외 시간 | 메뉴 변경, 업데이트 등 |
| 장애 복구 시간 | < 30분 | Docker 재시작으로 복구 |

### 4.2 데이터 백업

| 항목 | 방식 | 주기 |
|---|---|---|
| SQLite DB 파일 | 파일 복사 | 일 1회 (영업 종료 후) |
| 이미지 파일 | 디렉토리 복사 | DB 백업과 동시 |
| 백업 보관 | 로컬 또는 외부 스토리지 | 최근 7일분 보관 |

### 4.3 에러 처리

| 상황 | 처리 방식 |
|---|---|
| DB 연결 실패 | 자동 재연결 시도 (최대 3회) |
| SSE 연결 끊김 | 클라이언트 자동 재연결 (EventSource 기본 동작) |
| API 요청 실패 | 구조화된 에러 응답 (에러 코드 + 메시지) |
| 이미지 업로드 실패 | 트랜잭션 롤백, 임시 파일 정리 |
